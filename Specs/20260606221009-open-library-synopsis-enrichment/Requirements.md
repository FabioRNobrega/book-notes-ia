## Problem Statement

`BookContextService.GenerateContextAsync` builds the Ollama literary-context prompt using only `Book.Title` and `Book.Author`. The LLM must rely entirely on its own training data, which leads to hallucinated or inconsistent context paragraphs — particularly for less well-known titles. Open Library provides a free, two-step public API (search by title/author → fetch work details) that returns a real synopsis for many books. Incorporating that synopsis into the prompt gives the LLM factual grounding and makes the generated context more reliable. The synopsis must be fetched once and persisted to a new `Book.Synopsis` column so subsequent context regenerations and agent calls do not repeat the API round-trip.

## User Stories

- Given a book in my library has no stored context, when I click "Generate Context", then the system fetches the synopsis from Open Library, stores it on the book record, and uses it to produce a more accurate literary context paragraph.
- Given the agent calls `GenerateBookContext` for a book that has no stored context, when Open Library returns a synopsis, then the context generated by Ollama is grounded in that synopsis and the synopsis itself does not appear in the agent's tool response or conversation.
- Given the synopsis has already been fetched and saved to `Book.Synopsis`, when context is regenerated, then Open Library is not called again and the stored synopsis is used directly.
- Given Open Library has no synopsis for a book, when context generation runs, then the system falls back to the current pure-LLM behavior without error.
- Given Open Library returns an HTTP error or malformed JSON, when context generation runs, then the error is logged and generation proceeds without synopsis as a graceful fallback.

## Functional Requirements

1. **FR1** — `Book` model (`WebApp/Models/Book.cs`) must gain a new nullable `string? Synopsis` property. An EF Core migration must add the corresponding nullable `TEXT` column to the `book` table. Existing rows default to `NULL`.

2. **FR2** — A new `IOpenLibraryService` interface must expose a single method:
   ```csharp
   Task<string?> GetSynopsisAsync(string title, string author, CancellationToken ct = default);
   ```
   The interface must live alongside its implementation in `WebApp/Services/OpenLibraryService.cs`.

3. **FR3** — `OpenLibraryService` must implement `IOpenLibraryService` by making two HTTP calls in sequence:
   - `GET https://openlibrary.org/search.json?title={title}&author={author}` — reads `docs[0].key` as the work key.
   - `GET https://openlibrary.org{workKey}.json` — reads the `description` field.
   The implementation must support both the plain-string format (`"description": "..."`) and the object format (`"description": { "type": "/type/text", "value": "..." }`). It must return `null` when the search returns no docs, when the description field is absent, or when the description string is empty.

4. **FR4** — `OpenLibraryService` must catch all HTTP and JSON parsing exceptions, log them via `ILogger<OpenLibraryService>`, and return `null` to the caller without re-throwing. The caller must never observe an unhandled exception from this service.

5. **FR5** — `BookContextService.GenerateAndSaveAsync` (`WebApp/Services/BookContextService.cs`) must be updated to inject `IOpenLibraryService`. When `book.Synopsis` is null or whitespace, it must call `IOpenLibraryService.GetSynopsisAsync(book.Title, book.Author, ct)`. If a non-empty synopsis is returned, `book.Synopsis` must be assigned that value and persisted in the same `SaveChangesAsync` call that writes `book.Context`.

6. **FR6** — `BookContextService.GenerateContextAsync` must include the synopsis in the Ollama prompt when `book.Synopsis` is available. The synopsis must be presented as a factual grounding passage in the prompt and must not alter the ≤120-word plain-text output constraint or the existing prompt fields (author, book title, language, and content scope).

7. **FR7** — When `book.Synopsis` is null after the Open Library call (either not found or an error occurred), `BookContextService.GenerateContextAsync` must proceed with the existing pure-LLM generation behavior, producing a context paragraph without a synopsis in the prompt.

8. **FR8** — The synopsis must never appear in the tool result returned by `BookContextAgentTool` or in any agent-facing field. The tool response is `book.Context` only — unchanged from the current behavior.

9. **FR9** — `IOpenLibraryService` must be registered in `Program.cs` using `AddHttpClient<IOpenLibraryService, OpenLibraryService>()` so `HttpClient` is managed by the framework's `IHttpClientFactory`.

## Non-Functional Requirements

- **SOLID — Single Responsibility**: `OpenLibraryService` owns all HTTP interaction with the Open Library API. `BookContextService` owns context generation. No controller, agent tool, or EF Core model accesses `IOpenLibraryService` directly.
- **SOLID — Interface Segregation**: `IOpenLibraryService` exposes only `GetSynopsisAsync`. Callers that only need context generation must not depend on HTTP client configuration or Open Library data structures.
- **SOLID — Dependency Inversion**: `BookContextService` depends on `IOpenLibraryService`, not on `OpenLibraryService` or `HttpClient` directly.
- **No new runtime packages**: `System.Net.Http.Json` is available in the `Microsoft.AspNetCore.App` shared framework (.NET 9) and requires no new `PackageReference`.
- **User-data isolation**: `Book.Synopsis` is persisted on the user-scoped `Book` row and accessed only through the existing `UserId` predicate in `BookContextService`. No cross-user access is possible.
- **Testability**: `IOpenLibraryService` must be fakeable with a simple implementation (no real HTTP) so existing and new `BookContextServiceTests` can substitute it without network access.
- **No breaking changes**: `Book.Synopsis` is nullable; all existing books work without the column being populated. `BookContextAgentTool` and `BookContextController` are unchanged.

## Out of Scope

- Fallback to Google Books, Wikipedia, or any other external source when Open Library has no synopsis.
- Exposing `Book.Synopsis` in the Notes library view, Home page, or any Razor partial.
- Manual synopsis editing by the user.
- Periodic refresh of a stored synopsis when the Open Library record changes.
- Rate-limiting, retry policies, or circuit-breaker patterns beyond basic exception logging.
- Passing the synopsis as a standalone field to the agent or including it in orchestrator instructions.

## Open Questions

None. All design decisions were resolved during the discovery step:

- Synopsis is stored in `Book.Synopsis` (not fetched on every call).
- Both the user-triggered (`BookContextController POST /generate`) and agent-triggered (`BookContextAgentTool`) paths use the synopsis via `BookContextService.GenerateAndSaveAsync`.
- The synopsis is used only inside the Ollama prompt and is never surfaced to the agent.
- Tests mock `IOpenLibraryService`; no real HTTP integration tests are required for this spec.
