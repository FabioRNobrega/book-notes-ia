## Problem Statement

Today `WebApp/Program.cs` wires exactly one `AIAgent` (a `ChatClientAgent` built from a single Ollama-backed `IChatClient`) as a singleton, and `WebApp/Controllers/ChatController.cs` runs every chat turn through it via `IChatOrchestratorAgent`. There is no concept of multiple providers. `.env.example` already documents `AZURE_OPENAI_ENDPOINT` (`https://epoch-openai.openai.azure.com/`), `AZURE_LLM_DEPLOYMENT_NAME` (`epoch-gpt-4.1`), and `AZURE_OPENAI_API_KEY`, but nothing in the codebase reads or uses them yet. Users have no way to choose between a premium cloud model and the local free model, and `WebApp/Services/BookContextService.cs` (via `IOllamaService`) is hardcoded to Ollama for its own literary-context completion call, independent of any future agent choice.

## User Stories

- Given a signed-in reader on the home page, when they open the agent selector, then they see two options — "Premium — ChatGPT" and "Free — Qwen 3.5" — with the currently active one visibly indicated.
- Given a reader who selects "Premium — ChatGPT", when they send a chat message, then the response is generated by Azure OpenAI, the same tools (`GenerateBookContext`, `GetBookNotesWithAnalysis`, `GetRelevantBookNotes`) remain available, and the conversation context carries over from any prior turns in the session.
- Given a reader mid-conversation on "Free — Qwen 3.5", when they switch to "Premium — ChatGPT" without resetting the chat, then the next message is answered by Azure OpenAI using the same session history, and earlier messages remain visible and unaffected.
- Given a reader who has never chosen an agent, when they open the app, then "Free — Qwen 3.5" is active by default and clearly indicated as such.
- Given a reader on "Premium — ChatGPT" when the Azure OpenAI call fails (auth error, rate limit, timeout, network failure), then the chat surfaces the existing error message pattern and does not silently answer using the free model instead.
- Given a reader reviewing chat history, when they look at a past assistant message, then they can see which agent (Premium or Free) generated that specific response.

## Functional Requirements

1. FR1 — `WebApp/Views/Home/Index.cshtml` renders an `sl-select` agent picker (following the existing `sl-select`/`sl-option` pattern in `WebApp/Views/UserProfile/Upsert.cshtml`) with exactly two options: `premium` ("Premium — ChatGPT") and `free` ("Free — Qwen 3.5"). Any authenticated user may pick either option freely — there is no entitlement, billing, or permission check gating access to Premium.
2. FR2 — Changing the selector posts to a new `POST /chat/agent` endpoint on `ChatController`, which persists the choice (the plain string `"premium"` or `"free"`) scoped to the authenticated user in a new `activeagent:{userId}` cache key via `ICacheHandler`, using the same `SessionTtl` pattern as `activesessionid:{userId}`, without clearing `#chat`, the active session, or any cached context.
3. FR3 — `ChatController.Send` reads the user's active agent selection (defaulting to `"free"` when unset or invalid) and runs the chat turn through the matching `AIAgent` — an Azure OpenAI–backed `ChatClientAgent` for `"premium"`, the existing Ollama-backed `ChatClientAgent` for `"free"` — resolved through a new `IChatAgentProvider` abstraction (keyed by the same string) rather than a single constructor-injected agent.
4. FR4 — The `GenerateBookContext`, `GetBookNotesWithAnalysis`, and `GetRelevantBookNotes` `AIFunction` tools remain registered and callable identically by both agents; tool availability does not change based on the selected provider.
5. FR5 — `BookContextAgentTool`'s internal literary-context generation (`BookContextService.GenerateAndSaveAsync` → `GenerateContextAsync`) uses the chat client of whichever agent is currently selected for the turn (resolved via the same string key through a new `IChatClientProvider`), instead of being hardcoded to Ollama through `IOllamaService`.
6. FR6 — Switching the selected agent mid-conversation does not reset or reinitialize the serialized Microsoft Agent Framework session (`agentsession:{userId}:{sessionId}`); the next turn deserializes and continues the same session under the newly selected agent.
7. FR7 — Token/context-usage accounting (`TokenCountingChatClient`, the `agentcontext:{userId}:{sessionId}` cache entry, and the `<sl-progress-ring>` context meter) continues to function for both providers, reusing the existing `Ollama:NumCtx`-based percentage calculation as a shared denominator for this iteration.
8. FR8 — Each assistant-authored row in `WebApp/Models/ChatMessage.cs` records which agent produced it via a new `AgentType` string column (`"premium"` or `"free"`); both the live turn (`_BotMessage.cshtml`) and chat history (`Chat.cshtml`) render a small "Premium"/"Free" label alongside the existing response-time label for assistant messages. This is a plain string field for now, not a shared enum/struct — see [Out of Scope](#out-of-scope).
9. FR9 — A user who has never made a selection is treated as `"free"` by default, both for running chat turns and for the selector's initial displayed state — no explicit first-run action is required.
10. FR10 — If an Azure OpenAI call fails for any reason (authentication, rate limit, timeout, network), `ChatController.Send` returns the existing error partial (`_BotMessage` with `<p>Error: ...</p>`, per the current `catch` block) — there is no automatic fallback to the Free/Ollama agent for that turn.
11. FR11 — `HomeController.Index` reads the user's active agent selection (mirroring how it already reads `ContextUsagePct` from cache) into `ViewData`, so `Index.cshtml` can pre-select the correct `sl-option` and render a visible "currently active" indicator on page load, not just after a change event.

## Non-Functional Requirements

- Data isolation: `activeagent:{userId}` must be scoped by `ClaimTypes.NameIdentifier`, matching the existing `activesessionid:{userId}` / `agentsession:{userId}:{sessionId}` cache-key convention — no cross-user leakage of agent selection.
- SOLID / Open-Closed: provider resolution is isolated behind two narrow interfaces — `IChatAgentProvider` (returns `AIAgent`) and `IChatClientProvider` (returns `IChatClient`) — backed by string-keyed dependency injection, so `ChatOrchestratorAgent`, `BookContextService`, and `ChatController` depend on abstractions, not on `OllamaApiClient` or `AzureOpenAIClient` directly.
- Testability: both providers must be substitutable with fakes in unit tests; no test should require a reachable Ollama or Azure OpenAI endpoint (existing Docker-backed Postgres/pgvector integration tests remain the only tests allowed to hit real infrastructure).
- Security: `AZURE_OPENAI_API_KEY` must never be written to logs; `_logger.LogInformation` turn-stats logging in `ChatController.Send` must not be extended to include credentials or full Azure request/response payloads.
- Backwards compatibility: existing chat behavior for the Free/Ollama path (composer, mode buttons, reset, TTS playback, context ring, profile-driven instructions) must continue to work exactly as today — this feature adds a second path, it does not change the first.

## Out of Scope

- Embedding generation stays Ollama-only. `IEmbeddingService` / `IEmbeddingGenerator<string, Embedding<float>>` (backed by `mxbai-embed-large`) and pgvector-based book/note semantic lookup do not switch providers regardless of the selected chat agent.
- No new `UserProfile` database column or cross-device/long-term persisted preference. The active-agent selection lives in Redis with the same lifecycle as other per-user session cache state (`activesessionid:{userId}`).
- No formal `ChatAgentType` enum/struct in this iteration — agent selection is a plain string (`"premium"`/`"free"`) threaded through cache keys, DI keys, and the new `ChatMessage.AgentType` column. A shared, more formal representation is explicitly deferred to a future spec.
- No entitlement/billing/permission gate on Premium — every authenticated user may select it freely.
- No cost/latency disclaimer copy in the UI beyond the "Premium — ChatGPT" / "Free — Qwen 3.5" option labels themselves.
- No automatic fallback from Premium to Free when an Azure OpenAI call fails.
- No support for additional providers beyond exactly these two (Azure OpenAI / Ollama).
- No per-provider context-window sizing for the context-usage ring; both providers share the existing `Ollama:NumCtx`-based calculation for now.
- No changes to the TTS pipeline (`services/TtsService.Api`) — audio synthesis remains local/Supertonic regardless of the selected chat agent.
- No changes to Kindle import, book library, or notes upload flows beyond what is needed to keep `BookContextService` provider-aware.

## Open Questions

None — all scoping questions were resolved during discovery.
